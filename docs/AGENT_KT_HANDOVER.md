# Project KT / Handover Handbook (For Coders and Agents)

This is the canonical onboarding and handover document for this repository.  
Use it as the first read before making code changes.

---

## 1. Project Identity and Objective

- **Repository:** `AI-Revenue-Loss-Detection`
- **Domain:** fraud / revenue-loss detection
- **Primary datasets:**
  - **IEEE-CIS Fraud Detection** (tabular transaction + identity data)
  - **Elliptic Bitcoin dataset** (graph + tabular features)
- **Primary outputs:**
  - Reproducible fraud modeling pipeline (stages 1-5)
  - Streamlit dashboard for artifact-driven analysis
  - Report tables/metrics CSVs for manuscript/submission

In short: this project converts raw local datasets into cleaned features, model predictions, fusion metrics, graph experiments, and dashboard-visible artifacts.

---

## 2. High-Level Architecture

### 2.1 Pipeline layers

1. **Ingestion and preprocessing** (`stage01_data.py`)
2. **Classical ML + GBDT** (`stage02_gbdt.py`)
3. **Deep + anomaly channel** (`stage03_deep_anomaly.py`)
4. **Weighted fusion + report tables** (`stage04_fusion.py`, `report_tables.py`)
5. **Elliptic graph benchmarking** (`stage05_elliptic_graph.py`)

### 2.2 Runtime entrypoints

- CLI entrypoint: `main.py`
- Stage orchestrator: `src/fraud_ml/pipeline/run_all.py`
- Dashboard entrypoint: `app/app.py`

### 2.3 Package and path resolution

- Core package name: `fraud_ml`
- Path resolver: `src/fraud_ml/config/paths.py`
- Root discovery rule: repo root is the directory containing `DATASET_ieee-cis-elliptic/`
- Important contract: generated artifacts go to `processed_data/` and `figures/`

## 4. Environment and Dependencies

### 4.1 Python version policy

- Supported: **3.10.x to 3.12.x**
- Recommended: **3.11.x**
- Avoid: **3.13+** (TensorFlow wheel compatibility)

### 4.2 Main dependencies

- Data + ML: pandas, numpy, scikit-learn, scipy
- GBDT: lightgbm (preferred), xgboost (fallback path in Stage 2)
- Deep learning: tensorflow (preferred), sklearn MLP fallback in Stage 3
- Graph stage: PyTorch optional (required for true GCN run in Stage 5)
- Dashboard: streamlit + plotly
- Explainability: shap (optional in Stage 2)

### 4.3 Setup commands (Windows)

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\activate
python -m pip install --upgrade pip
pip install -r requirements.txt
pip install -e .
```

## 6. End-to-End Pipeline Behavior

## Stage 1 - Data Prep (`stage01_data.py`)

Purpose:

- Merge IEEE transaction + identity tables
- Missingness analysis, sparse-column dropping, imputation
- Feature engineering (`rel_day`, `rel_hour`, amount-ratio features)
- Generate EDA plots and cleaned datasets
- Prepare cleaned Elliptic transaction features

Typical outputs:

- `processed_data/ieee_train_merged_cleaned.csv`
- `processed_data/ieee_train_eda_ready.csv`
- `processed_data/preprocessing_config.json`
- `processed_data/ieee_missing_top20_summary.csv`
- `processed_data/elliptic_transactions_cleaned.csv`
- `figures/stage01_*.png`

## Stage 2 - Baselines + GBDT (`stage02_gbdt.py`)

Purpose:

- Train LR / DT / RF baselines
- Train main GBDT model (LightGBM preferred; XGBoost fallback)
- Optional SMOTE (`--smote`)
- Optional hyperparameter search (`--tune-gbdt`)
- Optional SHAP summary plot

Typical outputs:

- `processed_data/gbdt_preds.csv` (critical for Stage 3/4)
- `processed_data/stage02_experiment_config.json`
- `figures/stage02_gbdt_importance.png`
- `figures/stage02_shap_summary.png` (if SHAP available)

## Stage 3 - Deep + Anomaly (`stage03_deep_anomaly.py`)

Purpose:

- Build attention DNN branch (TensorFlow) and plain MLP baseline
- Fallback to sklearn `MLPClassifier` when TF is unavailable
- Train Isolation Forest anomaly channel
- Merge Stage 2 probability as a feature

Typical outputs:

- `processed_data/hybrid_dnn_anomaly_preds.csv` (critical for Stage 4)
- `processed_data/stage03_ieee_dnn_baselines.csv`
- `processed_data/stage03_experiment_config.json`
- `figures/stage03_*.png`

## Stage 4 - Fusion + Reporting (`stage04_fusion.py`)

Purpose:

- Join Stage 2 and Stage 3 score outputs
- Normalize channels and compute weighted hybrid score
- Tune threshold on validation split (best F1)
- Evaluate on test split and save final metrics
- Build manuscript report tables via `report_tables.py`

Default fusion weights:

- GBDT = `0.45`
- DNN = `0.40`
- Anomaly = `0.15`

Typical outputs:

- `processed_data/final_hybrid_comparison_metrics.csv`
- `processed_data/final_hybrid_scores.csv`
- `processed_data/final_hybrid_threshold.txt`
- `processed_data/report_table_1_ieee_cis.csv`
- `processed_data/report_table_2_elliptic.csv`
- `processed_data/report_table_3_ablation.csv`
- `processed_data/fusion_test_record_ids.csv`
- `processed_data/stage04_experiment_config.json`
- `figures/stage04_*.png`

## Stage 5 - Elliptic Graph (`stage05_elliptic_graph.py`)

Purpose:

- Temporal split evaluation on Elliptic
- Models include logistic regression, random forest, FraudGT-style MLP
- Optional 2-layer GCN (requires PyTorch)

Typical outputs:

- `processed_data/elliptic_graph_experiments.csv`
- `processed_data/stage05_experiment_config.json`
- `figures/stage05_*.png`

Skip behavior:

- Stage 5 may skip gracefully if required Elliptic inputs are missing
- Use `--skip-elliptic-graph` for full runs when graph stage is not needed

## 8. Split Strategy Semantics

Defined in `split_utils.py`:

- **random**: stratified random split for class balance (default)
- **temporal**:
  - IEEE: ordered by `TransactionDT` when available
  - Fusion stage: ordered by `transaction_dt`, `TransactionDT`, or `time_step` if present

Agent caution:

- Do not change split logic silently; it directly affects metric comparability with prior runs.

## 11. Known Failure Modes and Fixes

- `**ModuleNotFoundError: fraud_ml**`
  - Run from repo root and/or execute `pip install -e .`
- **TensorFlow import/install issues**
  - Use Python 3.11 and reinstall requirements
  - Stage 3 fallback (`MLPClassifier`) will run but gives different model behavior
- **Stage 4 missing required files**
  - Ensure Stage 2 and Stage 3 completed (`gbdt_preds.csv`, `hybrid_dnn_anomaly_preds.csv`)
- **Stage 5 skipped unexpectedly**
  - Verify `elliptic_transactions_cleaned.csv` exists (from Stage 1)
  - Verify Elliptic edge list file path is valid
  - Install PyTorch if GCN row is required
- **Dashboard shows empty sections**
  - Run corresponding stage(s) and verify artifacts in `processed_data/`
- **Large files accidentally staged**
  - Unstage and remove from index; validate `.gitignore` before commit

## 13. Testing and Verification Playbook

Minimum validation after any substantive pipeline change:

1. `python main.py --stage <affected_stage>`
2. If downstream impact exists, run next dependent stage
3. Confirm expected artifacts are generated/updated
4. Optional: open dashboard and verify affected page

For UI-only changes:

1. `python -m streamlit run app/app.py`
2. Validate modified page renders and handles missing artifacts gracefully

For split/model logic changes:

- Run with both `--split random` and `--split temporal` when relevant
- Check that config files (`stage0x_experiment_config.json`) reflect expected settings

## 14. Extension Guide (When Adding New Work)

If adding a new model or stage output:

1. Add generation logic in the appropriate `stage0x_*.py`
2. Save artifact to `processed_data/` with stable naming
3. Register/read it in dashboard (`app/components/file_utils.py` and target page)
4. Update `README.md` and this KT doc
5. Validate that old workflow still runs without the new artifact

If adding CLI flags:

1. Add parser option to `main.py`
2. Propagate to `run_all.py`
3. Thread into stage function signature
4. Document in README + KT


## 16. Definition of Done

A task is done only when:

- Requested scope is implemented
- Affected code paths execute without new errors
- Artifacts/columns expected by downstream consumers are preserved (or intentionally migrated)
- Documentation is updated for behavior/command changes
- No forbidden large/generated files are staged for commit
